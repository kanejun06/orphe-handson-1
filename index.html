⚡ 当日までにやってくること（所要15分・重要）

当日のWS時間は実質で90分しかありません！下の3つを事前に各自のPCで済ませてくると、本番は「作る」だけに集中できます。

  • ① Google Chrome（または Edge）を入れてくる。Safari・Firefox では ORPHE につながりません。ふだん別のブラウザを使っている人も、当日は Chrome で開いてください。
  • ② p5.js Web Editor のアカウントを作る。editor.p5js.org/signup から登録（無料）。作品の保存・共有に使います。
  • ③ Teams の Copilot を一度開いて、何か質問してみる。下の「Copilotってなに？」を読んで、実際に1往復だけ会話しておくと当日スムーズです。

事前共有Copilot ってなに？　— 今回の主役です

このWSでは、プログラムを 自分で1行ずつ書きません。代わりに Copilot（AIアシスタント）に日本語で「こうしたい」と伝えて、コードを書いてもらうやり方（＝バイブcoding）で進めます。Copilot は みなさんの Teams の中にすでに入っています。新しくインストールするものはありません。

Copilot にできること（このWordで使うこと）

  • コードを書いてもらう。「足を振ったら音が鳴るようにして」と頼むと、プログラムを書いて返してくれます。
  • エラーを直してもらう。赤いエラー文を貼って「直して」と言えば、原因と直し方を教えてくれます。
  • コードの意味を説明してもらう。「このコードが何をしているか教えて」で、日本語で解説してくれます。
  • アイデア出しの相手になってもらう。「こんなセンサーで作れる作品を3つ提案して」のように壁打ちできます。

Teams での開き方

  1. Teams を開き、左側のメニュー（または上部の検索バー横）にある Copilot を選ぶ。
  2. 下の入力欄に、ふつうにチャットするように日本語で書いて送る。
  3. 返ってきた文章やコードは、右上のコピーボタンで丸ごとコピーできる。
💡 まず1回ためしてみよう（事前課題）
Teams の Copilot に、こう送ってみてください ——「p5.js って何ができるツール？ プログラミング初心者にもわかるように3行で教えて。」。返事が来れば準備OK。これが当日やる「話しかけ方」の練習になります。
⚠ Copilot を使うときの心がまえ
AIは便利ですが、間違ったコードや動かないコードを返すこともあります。鵜呑みにせず「動かなかったらまた頼む」を繰り返すのが普通です。そして 「なぜそう作ったか」を説明するのは自分自身。出てきたコードは、Copilot に意味を聞いて自分でも理解しておきましょう。

BASICvibe coding の基本 ／ Copilotの話しかけ方

Copilotへの「話しかけ方」

この先のBLOCK1〜3には、各作業でCopilotに送るお手本プロンプトがそれぞれ載っています。その前に、共通のコツだけ押さえておきましょう。バイブコーディングは、技術ではなく「何を作りたいか」を言葉にするのが本体です。

POINTうまく頼む4つのコツ

  1. 作りたい体験を先に言う。「足踏みすると画面の花が増える」のように、技術ではなく体験で。
  2. 土台のコードを一緒に貼る。「このコードに追加して」と渡すと的外れが激減。
  3. 一度に1つだけ頼む。色も音もではなく、まず色、動いたら次に音。
  4. エラーもそのまま伝える。赤い文字をコピペして「このエラーを直して」。

BLOCK 1つなぐ ／ スターターを貼って接続 + 役割分担

まずは「つながって遊べた！」を全員で

出発点は、菊川さん作のスターターHTMLを貼ること。接続UI・グラフ・サンプルゲームが最初から動くので、つないで遊ぶところまで一気に行けます。ここが通れば、あとは改造するだけです。

ORPHEORPHE CORE とは

足に着ける小さなセンサーです。加速度・角速度・向き（オイラー角）・歩数・踏み込みなどをリアルタイムで取れます。これを p5.js プログラムの「入力」に使います。Web Bluetooth でブラウザから直接つなぎます。

✦ORPHE-CORE.js 公式（サンプル多数） ▶p5.js で始める Quick Start 📖基礎チュートリアル（もっと知りたい人向け）

仕組み接続まわりは「全部おまかせ」でOK

このスターターは CoreToolkit.js という仕組みを使っていて、接続・再接続・センサー値の受信を全部自動でやってくれます。だから学生は「どうやって繋ぐか」を書く必要がありません。覚えることはこれだけです。

  • 画面の接続ボタンはスターターが最初から表示します（自分で作らない）。
  • センサーの値は sensors[0]（CORE 01）から読みます。例：sensors[0].euler.pitch（向き）、sensors[0].acc.x（加速度）、sensors[0].steps（歩数）。
  • 新しく new Orphe() を作ってはいけません（接続が二重になって壊れます）。
💡 つまり学生がやることは
「sensors[0] に入っている値を、ゲームの動きにどう結びつけるか」を考えるだけ。そこをCopilotに頼んで改造していきます。読めるデータの一覧はページ下部の早見表に。

STEP 1貼って、つないで、遊んでみる（最重要）

このWSでは、ORPHE開発者の菊川さんが用意したスターターHTMLを出発点にします。接続UI・センサーのグラフ・サンプルゲーム（モーションピンポン）が最初から全部入っていて、貼るだけで動きます。ゼロからコードを書く必要はありません。まずは改造せず、そのまま動かして接続を確認しましょう。CORE 1台でも、もう片方は自動でAIが操作するのでゲームになります。

  1. 下のボタンから orphe_starter.html をダウンロードして開く（中身は触らなくてOK）。
  2. p5.js Web Editor を開き、左の index.html を選ぶ。
  3. index.html の中身を全部消して、スターターHTMLファイルをドラッグ＆ドロップする。
  4. ▶ Run すれば、接続ボタンとゲーム画面が出ます。これがスタート地点です。
  5. 画面上部の 「CORE 01」の接続ボタンを押し、ORPHE CORE を選ぶ（Chromeで）。
  6. ステータスが connected になったら、足（センサー）を動かすと左パドルが動きます。右パドルはAIが担当。ラリーが続けば接続成功です。
⬇orphe_starter.html をダウンロード ◷中身を別タブで見る
💡 これは「全部入り1ファイル」です
p5.js Web Editor は通常 index.html / sketch.js / style.css の3ファイル構成ですが、このスターターは index.html 1枚に全部書かれています。だから sketch.js は使わず、編集はすべて index.html の中で行います。
⚠ つながらない時はまずこの4つ
  • ① Chrome / Edge で開いているか（Safari・Firefoxは不可）。
  • ② p5.js Web Editor は「▶ Run」した後、プレビュー画面内のボタンから接続。それでもダイアログが出なければ、画面を新しいタブで開き直す。
  • ③ 接続は必ず画面のボタンを自分でクリックして開始（自動接続はブラウザの仕様でできません）。
  • ④ ORPHE本体の電源がONか、他のPCと接続済みになっていないか確認。
  • ⑤ CORE未接続でも、キーボードの↑↓キーで左パドルを動かして動作確認できます。

仕組み中身は「セーフエリア」と「アプリケーションエリア」に分かれています

スターターHTMLは、大きく セーフエリア（接続まわり） と アプリケーションエリア（アプリ本体） の2つに分かれています。接続やセンサーの受信は セーフエリア が自動でやってくれるので、改造するのはアプリケーションエリアだけ。細かい仕組みを覚える必要はありません。

定型ヘッダー文すべてのお願いの先頭に貼る（全STEP共通）

このスターターを安全に改造させるための前提文です。Copilotに改造を頼むときは、まずこれを先頭に貼ってください。「セーフエリア（接続まわり）は触らせない・アプリケーションエリアだけ直す・全文で返す」を守らせます。

header_prompt.txt
# 前提条件（必ず守ってください）
- これは ORPHE CORE + CoreToolkit.js のスターターHTML（1ファイル）です。
  これから渡すHTML全体をベースに改造してください。
- 編集してよいのは COPILOT EDIT AREA 1・2・3 だけです。
  SAFE FOUNDATION AREA（接続・センサー受信・グラフ・クォータニオン処理）は
  絶対に変更・削除しないでください。
- 新しく new Orphe() を作らないでください。接続は CoreToolkit が
  bles[0] / bles[1] で既に用意しています。
- toolkit0, toolkit1, gameCanvas, chartAcc0 などのIDや、
  setupCoreToolkit(), attachSensorCallbacks(), setupCharts(),
  normalizeQuaternion() の名前は変えないでください。
- センサー値は次から読めます（自分で取得処理を書かない）：
  sensors[0] が CORE 01、sensors[1] が CORE 02。
  sensors[id].euler.pitch / roll / yaw（向き）
  sensors[id].acc.x / y / z（加速度）
  sensors[id].gyro.x / y / z（角速度）
  sensors[id].steps（歩数）
  sensors[id].gait / stride / footAngle / pronation / landingImpact（歩行解析）
- 回答は、改造後のHTMLを【全文】で出してください。
  「（省略）」などを使わず、そのまま index.html に貼れる完成形で。
- ブラウザは Chrome 想定（Web Bluetooth）。
- コードは初心者が読めるよう、変更点に日本語コメントを付けてください。
BLOCK 2観察する ／ 内蔵グラフで値を見る

どの動きがどの値に効くかを見つける

数字を「見える化」すると、企画のアイデアが湧いてきます。足を動かしたとき、どの値がどう変わるかを観察しましょう。

STEP 2グラフで「どの動きが効くか」を見つける

スターター画面の下に、加速度・角速度・向き（Euler）のリアルタイムグラフが最初から表示されています。コードを書かなくても、ここを見ながら足を動かせば「どの値がどう反応するか」がすぐ分かります。これが企画の発想源になります。

  1. CORE を足に着けて、歩く・つま先を上げる・かかとで踏む・ジャンプを試す。
  2. そのとき どのグラフ（acc / gyro / euler）が大きく動くかを観察する。
  3. 上のステータス欄の steps（歩数）や acc の数値も見る。
💡 観察のポイント
「強く踏むと acc が跳ねる」「つま先を上げると euler の pitch が変わる」など、動き → 反応する値の組をオブザーバーがメモしておくと、BLOCK3で「何を操作に使うか」が決めやすくなります。読めるデータの一覧はページ下部の早見表に。
BLOCK 3動きを変える ／ ビジュアル・音・スコア

動きを「何か」に結びつける

値を、目に見える変化・音・スコアに変換します。ここがグループのアイデアを入れる場所。1機能ずつ足すのがコツです。

STEP 3操作を変えてみる（アプリケーションエリア）

まずは小さな改造から。「何で操作するか」を変えます。今は「向き（pitch）」でパドルが動きますが、これを別のセンサー値に変えてみましょう。定型ヘッダー文＋スターターHTML全文を貼ったうえで、下を送ります。

step3.txt
（先頭に定型ヘッダー文＋スターターHTML全文を貼ったうえで）

COPILOT EDIT AREA 2 の sensorToPaddleY() だけを変えて、
パドルを「向き(pitch)」ではなく「加速度の上下(acc.y)」で
動かすようにしてください。
動きが急すぎる/鈍いときに調整できるよう、感度の数値も
分かりやすくコメントしてください。
SAFE FOUNDATION AREA は変更しないでください。
💡 調整もCopilotに頼める
「反応が鈍い／敏感すぎる」ときは、そのまま「感度を上げて／下げて」と頼めばOK。BLOCK2のグラフで見た“よく動く値”を操作に使うと、気持ちよく動きます。

STEP 4自分たちのゲームに作り替える（アプリケーションエリア）

いよいよ企画を形に。見た目や難易度、ゲームのルールそのもの（どちらもアプリケーションエリア）を変えます。欲張らず、まず一番簡単な1つから。

step4.txt
（先頭に定型ヘッダー文＋いま動いているHTML全文を貼ったうえで）

COPILOT EDIT AREA 1・2・3 だけを変えて、
〇〇〇〇（作りたいゲームを一言で）にしてください。
まず △△△△ の部分だけ変えて、他はできるだけ残してください。
SAFE FOUNDATION AREA（接続・センサー・グラフ）は触らないでください。
改造後のHTMLを全文で出してください。
💡 作り替えの例
「ボールを避けるゲームにして」「踏み込んだら主人公がジャンプするゲームにして」「スコアと制限時間を足して」など。1回に1つの変更にすると、動かなくなっても戻しやすいです。

BRAINSTORM企画を1つに決める（Copilotと相談）

作るものに迷ったら、Copilotを壁打ち相手にできます。下を送ってアイデアを出させ、そのなかから1つに絞ってBLOCK3の制作へ進みます。

brainstorm.txt
足に着けるセンサーで、加速度・向き・歩数・踏み込みが取れます。
これを使った、90分で作れる簡単な作品アイデアを
「ゲーム」「ビジュアル」「音」「測定・気づき」の4方向で
3つずつ、各1行で出してください。実装が重いものは避けてください。

出た案は必ず「いちばん簡単に試せる1つ」に絞ってから作り始めること。欲張ると90分で動きません。

APPENDIX付録 ／ リファレンス

スターターで読めるデータ早見表

スターターでは、接続したセンサーの値が sensors[0]（CORE 01）にいつも入っています。下の「読み方」をそのままCopilotに伝えると、ピンポイントで正しい改造が返ってきます。「パドルを歩数で動かして → sensors[0].steps を使って」のように。

取りたいものコードでの読み方
向き（傾き pitch/roll/yaw）sensors[0].euler.pitch / .roll / .yaw
加速度（振り・衝撃）sensors[0].acc.x / .y / .z
角速度（回転の速さ）sensors[0].gyro.x / .y / .z
歩数sensors[0].steps
歩容（歩く/走るなど）sensors[0].gait
ストライド（歩幅）sensors[0].stride
足の角度（接地角）sensors[0].footAngle
プロネーション（内/外への倒れ）sensors[0].pronation
着地の衝撃sensors[0].landingImpact

※ 2台目をつないだ場合は sensors[1] で同じように読めます。歩容・ストライド・足角度などは STEP_ANALYSIS のデータが届いたときに入ります（スターターは両方受信する設定なのでそのままでOK）。
